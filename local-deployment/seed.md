# Тестовый ресторан для загрузки в БД

## Важно: ID сущностей

**Все ID в этом файле будут отрицательными (минусовыми)**, например: -1, -2, -15, -100 и т.д. Это необходимо для того, чтобы тестовые данные не конфликтовали с реальными данными в БД.

## Примечания для создания SQL скрипта

1. **Порядок вставки данных:**
   - Сначала ресторан (restaurants)
   - Затем блюда меню (menu_items) - используя SELECT для получения ID категорий по названию
   - Затем этажи (floors)
   - Затем залы (rooms)
   - Затем столы (tables)
   - Затем акции (promotions) - используя SELECT для получения ID типов по коду

2. **Категории меню:** Категории меню уже создаются миграциями в admin-api (V5__insert_menu_categories.sql). В SQL скрипте нужно использовать подзапрос для получения ID категории по названию:
   ```sql
   menu_category_id = (SELECT id FROM menu_categories WHERE name = 'Паста')
   ```

3. **Типы промо-акций:** Типы промо-акций уже создаются миграциями в admin-api (V10__insert_promotion_types.sql). В SQL скрипте нужно использовать подзапрос для получения ID типа по коду:
   ```sql
   promotion_type_id = (SELECT id FROM promotion_types WHERE code = 'DISCOUNT')
   ```

4. **Изображения:** Все image_id указаны как отрицательные, но сами изображения будут загружены через админку позже. В SQL скрипте можно либо оставить их как NULL, либо создать заглушки в таблице images с соответствующими отрицательными ID.

5. **Даты:** В промо-акциях используются даты 2024 года. При создании скрипта можно использовать текущую дату или даты относительно текущей даты.

6. **BaseEntity поля:** Не забыть про поля `is_active`, `created_at`, `updated_at`, `deleted_at` (где применимо).
